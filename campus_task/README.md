## 一、项目简介

CampusTask 是一个极简的命令行校园任务管理工具。学生可以通过命令行添加、查看、完成课程作业、实验报告、考试复习等任务。数据以 JSON 格式保存，程序退出后再打开数据不会丢失。

**版本**：v0.3.0（搜索、逾期检测、优先级、截止日期）

---

## 二、环境要求

- Python 3.8 及以上
- 操作系统：Windows / macOS / Linux
- 无需安装第三方依赖（仅使用 Python 标准库）

---

## 三、使用方法

```bash
# 查看帮助
python main.py

# 添加任务（可选 --deadline 和 --priority）
python main.py add "完成软件工程实验1"
python main.py add "复习高数" --deadline 2026-06-20 --priority high

# 列出所有任务（可选 pending / done）
python main.py list
python main.py list pending

# 完成任务
python main.py done 1

# 搜索任务
python main.py search 软件工程

# 查看逾期任务
python main.py overdue
```

## 四、数据文件说明

任务数据保存在 `tasks.json` 中，格式如下：

```json
[
  {
    "id": 1,
    "title": "完成软件工程实验1",
    "status": "done",
    "created_at": "2026-06-14 10:19:00"
  },
  {
    "id": 2,
    "title": "复习高等数学第三章",
    "status": "pending",
    "created_at": "2026-06-14 10:19:03"
  }
]
```

每个任务包含四个字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | 整数 | 任务编号，自动递增 |
| `title` | 字符串 | 任务标题 |
| `status` | 字符串 | 任务状态：`pending`（未完成）或 `done`（已完成） |
| `created_at` | 字符串 | 创建时间，格式 `YYYY-MM-DD HH:MM:SS` |

## 六、用户故事与验收标准

基于采访结果，整理出以下 5 条用户故事：

---

### 用户故事 1：添加任务

> **作为** 一名学生，  
> **我希望** 用一条命令快速添加课程任务（如"完成软件工程实验1"），  
> **以便** 随时记录待办事项，不会遗忘作业和报告。

**验收标准：**
- [ ] 输入 `python main.py add "任务标题"`，任务成功添加到列表
- [ ] 系统自动生成不重复的编号
- [ ] 每条任务记录创建时间
- [ ] 新建任务的状态默认为"未完成"
- [ ] 数据保存到 tasks.json，程序重启后仍然存在

---

### 用户故事 2：查看任务列表

> **作为** 一名学生，  
> **我希望** 用一条命令查看当前所有任务及其状态，  
> **以便** 快速了解还有哪些作业没交、哪些已完成。

**验收标准：**
- [ ] 输入 `python main.py list`，显示所有任务
- [ ] 列表包含编号、标题、状态、创建时间
- [ ] 已完成任务和未完成任务有明显区分（如不同图标）
- [ ] 任务列表为空时给出友好提示（如"暂无任务"）

---

### 用户故事 3：标记任务为完成

> **作为** 一名学生，  
> **我希望** 用任务编号将任务标记为"已完成"，  
> **以便** 跟踪自己的进度，获得完成任务的成就感。

**验收标准：**
- [ ] 输入 `python main.py done <编号>`，对应任务状态变为"done"
- [ ] 对不存在的编号给出明确错误提示
- [ ] 对已经完成的任务再次标记时，提示用户"已经是完成状态"
- [ ] 状态变更立即持久化到 tasks.json

---

### 用户故事 4：数据持久化

> **作为** 一名学生，  
> **我希望** 任务数据在程序关闭后不会丢失，  
> **以便** 可以在任何时候继续管理任务，不需要重复记录。

**验收标准：**
- [ ] 执行添加和完成任务后，tasks.json 实时更新
- [ ] 程序关闭后重新运行 `list`，数据完整保留
- [ ] 若 tasks.json 文件不存在，程序自动创建空列表而不崩溃
- [ ] 若 tasks.json 文件内容损坏，程序给出警告并重置

---

### 用户故事 5：简洁直观的命令行体验

> **作为** 一名计算机专业学生，  
> **我希望** 命令行操作简洁直观，输入错误时得到清晰提示，  
> **以便** 在编码间隙快速管理任务而不打断思路。

**验收标准：**
- [ ] 无参数运行 `python main.py` 显示帮助信息
- [ ] 输入错误命令时提示"未知命令"并显示正确用法
- [ ] `add` 缺少标题时提示用户补充
- [ ] `done` 缺少编号或编号非数字时给出具体错误提示
- [ ] 帮助信息清晰展示所有可用命令

## 八、课后反思

### 问题：为什么"会写Python"还不等于"会做软件"？

**我的理解：**

"会写Python"只是掌握了编程语言的语法和基本库，相当于学会了使用锤子和锯子。而"做软件"需要面对的是一个完整的问题解决链条：

1. **需求不是代码**：在这次实验中，我最大的收获是理解了"需求分析"的价值。如果直接写代码，我可能只会做一个"能加任务、能完成任务"的程序。但通过采访同学，我才发现学生真正的问题是"在编码间隙快速操作"、"数据不能丢失"、"空列表要给提示而不是报错"——这些都不是Python语法能告诉我的。

2. **软件=功能+边界处理+用户体验**：会写Python可以写出 `task["status"] = "done"`，但做软件要考虑：如果用户输入了不存在的编号怎么办？如果JSON文件被手动改坏了怎么办？如果用户什么都不输入直接回车怎么办？这些"如果"才是软件工程的核心。

3. **可验收性至关重要**：实验要求写"验收标准"，这让我意识到模糊的需求（"做个任务管理工具"）和可验收的需求（"输入done 1后，任务1的状态变为done，且tasks.json随之更新"）之间的巨大差距。写完代码后对照验收标准逐条检查，才能确认"做完了"。

4. **软件是为他人而写**：写Python练习题只要自己看懂就行。但做软件要考虑别人（用户）怎么用——他们可能不按你预期的顺序操作，可能输入奇怪的内容。为他人设计，意味着需要站在使用者的角度思考，而这比写代码本身更难。

5. **工程是持久的**：写完一个脚本只要几分钟，但后续的维护、扩展、Bug修复可能需要几天甚至几周。实验一的程序虽然简单，但结构化程度直接影响后面实验二的重构难度。

**总结**：会写Python是技术基础，会做软件是工程思维。前者让你能写出代码，后者让你能写出**可用的、可靠的、可持续维护的代码**。两者之间横亘着一座桥，桥的名字叫"软件工程"。
