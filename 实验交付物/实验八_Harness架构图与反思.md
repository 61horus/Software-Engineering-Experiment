# 实验八 — AI Harness 架构图

```
用户自然语言输入
        │
        ▼
┌─────────────────┐
│ prompt_builder   │  构造提示（含当前任务状态）
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  mock_model      │  模拟AI生成意图(action + args)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ parse_model_out  │  解析模型输出为标准字典
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   guardrail      │  安全检查（如"删除所有"需人工确认）
└────────┬────────┘
         │ 通过
         ▼
┌─────────────────┐
│  tool_executor   │  调用 CampusTask 功能执行
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  trace_logger    │  全流程写入 trace.jsonl
└─────────────────┘

          评测回路:
          eval_cases.json → run_eval() → 准确率
```

## 各模块说明

| 模块 | 职责 | 人/模型 |
|------|------|---------|
| prompt_builder | 组合用户输入+任务状态 | 人写代码 |
| mock_model | 规则匹配生成意图 | 模拟模型 |
| parser | 标准化输出格式 | 人写代码 |
| guardrail | 高风险操作拦截 | 人写代码 |
| tool_executor | 调用 CampusTask API | 人写代码 |
| trace_logger | JSONL 全流程记录 | 人写代码 |
| eval_runner | 评测准确率 | 人写代码 |

## 反思：AI Harness 工程与软件工程的共性

1. **仍然需要接口设计** — prompt_builder 的输出格式是 mock_model 的输入契约，格式不匹配整个链条断裂
2. **测试比传统代码更重要** — 模型输出不确定，eval_cases 是唯一的质量锚点
3. **日志是唯一的事后证据** — trace.jsonl 记录了每次运行的完整决策链
4. **Guardrail = 权限控制** — "删除所有任务"需要人工确认，这和传统系统的确认对话框原理相同
5. **人写代码 vs 模型输出** — 管道（prompt→parse→guard→exec）仍然由人编码，模型只负责"意图判断"这一环

**当模型可能出错时，需求、测试、日志、权限和人工审批反而更重要** — 因为你不信任模型，所以必须用工程手段兜底。
